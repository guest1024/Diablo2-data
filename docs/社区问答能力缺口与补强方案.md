# Diablo II 社区问答能力缺口与补强方案

这份文档专门对照你提出的目标来回答：

> 未来面向社区问答，只要涉及到游戏攻略的检索都尽可能满足能力。

重点不是再讲抽象架构，而是明确：

1. **当前已有了什么**
2. **还缺什么**
3. **下一步该优先补哪一层数据 / 能力**

---

## 1. 先讲结论

当前仓库已经具备：

- 大规模基础语料
- 中英术语映射
- curated anchor 稳定召回层
- primary / curated 双路路由
- GraphRAG 设计文档
- 验证闭环

但如果目标是：

> “社区里的问答，只要涉及到游戏攻略的检索都可以做”

那现在还差三件关键事：

1. **结构化游戏规则层还不够深**
2. **Build / 攻略 / 任务打法层还不够系统**
3. **图谱层还缺正式可导入的 alias-registry / term-equivalence / build-archetype 支撑资产**

---

## 2. 对照你的六大百科模块，当前缺口是什么

## 2.1 物品与装备系统

### 当前已有

- Base / Unique / Set / Runeword 资料已经较强
- 中英俗称映射已经覆盖大量高频问法
- curated anchor 已覆盖大量高频装备黑话

### 当前缺口

- 变量范围还没有完全结构化成统一事实层
- `BaseItem -> 最大孔数 / 需求属性 / 可做哪些符文之语`
  还没有做成统一 graph facts
- 护身符 / 珠宝 / 手工公式 还缺更细粒度 registry

### 优先补法

1. 把 `blizzhackers/d2data` 重接入为结构化事实源
2. 补：
   - `base-item facts`
   - `runeword requirements`
   - `unique variable ranges`
   - `crafted recipe facts`

---

## 2.2 游戏核心机制

### 当前已有

- 已有一部分 FAQ / 机制说明文本
- 已有 mixed retrieval 能力

### 当前缺口

- Breakpoints（FCR/FHR/FBR/IAS）仍未正式结构化
- 命中率 / 格挡率 / 抗性 / DR / 吸血惩罚等公式缺少 calculator-ready 层
- Drop / TC / qlvl / mlvl / MF 递减曲线缺少专门事实层

### 优先补法

建议新增：

- `breakpoints.jsonl`
- `combat-formulas.jsonl`
- `drop-mechanics.jsonl`

并在图谱里挂：

- `USES_BREAKPOINT`
- `AFFECTED_BY_FORMULA`
- `DROP_RULE`

---

## 2.3 职业与流派

### 当前已有

- 已经开始补 build archetype：
  - 锤丁
  - 冰法
  - 电法
  - 新星电法
  - 标马
  - 弓马
  - 狼德
  - 陷阱刺客
  - 召唤死灵

### 当前缺口

- build 还没有完整拆成：
  - 技能加点
  - 装备优先级
  - 佣兵配置
  - 开荒 / 毕业 / 土豪 / 平民分层
- 还没有正式 `build-archetypes.jsonl` 生产化使用

### 优先补法

建议下一步把 build schema 固定成：

```json
{
  "build_id": "build::hammerdin",
  "class": "Paladin",
  "aliases": ["锤丁", "Hammerdin"],
  "core_skills": ["Blessed Hammer", "Concentration"],
  "gear_core": ["Enigma", "Heart of the Oak", "Arachnid Mesh"],
  "merc_core": ["Insight", "Holy Freeze Merc"],
  "farm_targets": ["Chaos Sanctuary", "Baal"],
  "phase": "starter|midgame|endgame"
}
```

## 3. 对社区问答 Bot 架构来说，现在缺什么

## 3.1 缺一个“问题类型路由层”

当前检索底座已经有了，但社区问答真正上线时还要分流：

### 精确词典类

- 无限怎么做
- 乔丹是什么
- 安头是什么

=> 走 `term map + curated + structured facts`

### 机制计算类

- 法师上 105 FCR 还差多少
- 格挡怎么计算

=> 走 `formula / breakpoint calculators`

### 攻略建议类

- 冰法怎么开荒
- 锤丁怎么打 Uber

=> 走 `guide chunks + build schema + rerank`

## 3.2 缺一个“source-aware answer policy”

未来社区问答一定要区分：

- 精确事实：优先 official / structured_db
- 攻略经验：优先 guide / wiki / curated build
- 争议问题：必须显示来源和版本

## 3.3 缺一个“失败问答回流”机制

建议后续记录：

- 哪些 query 命中了错误 route
- 哪些 query 召回不够
- 哪些 query 需要新 alias

## 6. 最终一句话判断

### 你当前的数据和架构，适合做什么？

适合做：

- 百科词典问答
- 装备 / 技能 / 场景 /黑话映射问答
- 中英混合实体检索
- 初步 GraphRAG

### 你当前还不够强的是什么？

还不够强的是：

- 机制计算
- build 结构化回答
- 佣兵体系结构化
- 掉落 / TC / patch 差异精确问答

### 下一步该做什么？

> **把“文本知识库”继续升级为“结构化知识底座 + 检索友好图谱 + 攻略型 schema”三层体系。**
