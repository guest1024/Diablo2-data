# 最小问答原型设计（Phase 1）

这份文档把第一阶段“**先跑通一个最小可用问答原型**”拆成可直接开发的模块、接口和输入输出。

当前约束：

- 数据主入口：`docs/tier0/merged/`
- 向量数据库：`ChromaDB`
- 图数据库：`Neo4j`
- 主规范：`docs/data-spec-v1.md`

---

## 1. 原型目标

先做一个能回答以下问题的最小系统：

- Spirit 是什么？
- 军帽是什么？
- Hellfire Torch 有什么用？
- 某个技能是什么？
- 某个区域适合干什么？

输出必须包含：

- 结论
- 至少 1 个来源 URL
- 至少 1 段证据文本

不要求一开始就：

- 支持复杂多跳推理
- 做极致版本差异判断
- 做完整 build 推荐

---

## 2. 最小系统模块

Phase 1 只做 5 个模块：

1. `data_loader`
2. `alias_resolver`
3. `chunk_retriever`
4. `grounding_builder`
5. `answer_generator`

---

## 3. 模块设计

## 3.1 data_loader

### 输入
- `docs/tier0/merged/canonical-entities.jsonl`
- `docs/tier0/merged/canonical-claims.jsonl`
- `docs/tier0/merged/provenance.jsonl`
- `docs/tier0/merged/chunks.jsonl`
- `docs/chroma-ready/chunks.jsonl`

### 输出
- 内存索引 / 本地缓存对象

### 责任
- 加载 JSONL
- 建立：
  - `entity_by_id`
  - `alias_index`
  - `claims_by_subject`
  - `provenance_by_claim`
  - `chunk_by_doc_or_chunk_id`

---

## 3.2 alias_resolver

### 输入
```json
{
  "query": "精神盾底材是什么"
}
```

### 输出
```json
{
  "language": "zh",
  "normalized_query": "精神盾底材是什么",
  "resolved_entities": [
    {
      "canonical_id": "Runeword:spirit",
      "score": 0.97,
      "match_type": "alias_exact"
    }
  ]
}
```

### 责任
1. query normalization
2. alias exact match
3. alias fuzzy match
4. 如果没命中，再回退到 embedding/entity lookup

### Phase 1 目标
先只做：
- exact
- contains
- 小范围 fuzzy

---

## 3.3 chunk_retriever

### 输入
```json
{
  "query": "精神盾底材是什么",
  "resolved_entities": ["Runeword:spirit"]
}
```

### 输出
```json
{
  "candidate_chunks": [
    {
      "chunk_id": "...",
      "score": 0.91,
      "source_id": "diablo2-io"
    }
  ]
}
```

### 责任
使用三路召回：

1. alias/实体约束
2. lexical/BM25
3. Chroma 向量召回

### Phase 1 最简实现
- 先做：
  - entity-filtered lexical
  - Chroma dense retrieval

---

## 3.4 grounding_builder

### 输入
- entity
- claims
- provenance
- top chunks

### 输出
```json
{
  "facts": [...],
  "sources": [...],
  "evidence_chunks": [...]
}
```

### 责任
- 从 `canonical_claims` 找核心事实
- 从 `provenance` 找来源
- 从 `chunks` 找正文证据

### Phase 1 只要做到
- 每次回答都能带：
  - canonical entity
  - 1~3 个 claims
  - 1~3 个 evidence chunks
  - 1~3 个 source URLs

---

## 3.5 answer_generator

### 输入
```json
{
  "query": "...",
  "entity": {...},
  "facts": [...],
  "sources": [...],
  "evidence_chunks": [...]
}
```

### 输出
```json
{
  "answer": "...",
  "sources": [...],
  "evidence_chunk_ids": [...]
}
```

### 责任
- 组织最终 prompt
- 让 LLM 基于 facts + chunks 生成答案
- 输出带引用

### Phase 1 模板建议

结构固定为：

1. 结论
2. 关键事实
3. 证据来源
4. 版本说明（如果有）

---

## 4. API 设计

推荐 Phase 1 只做一个最小接口：

## `POST /qa`

### 请求
```json
{
  "query": "精神盾底材是什么"
}
```

### 响应
```json
{
  "query": "精神盾底材是什么",
  "language": "zh",
  "resolved_entities": [...],
  "answer": "...",
  "sources": [...],
  "evidence_chunks": [...]
}
```

---

## 5. ChromaDB 接入方式

## collection
- `evidence_chunks`

### 导入来源
- `docs/chroma-ready/chunks.jsonl`

### 字段
- `id`
- `content`
- `metadata`

### metadata 至少包含
- `doc_id`
- `source_id`
- `language`
- `authority_tier`
- `game_variant`
- `version_tokens`
- `entity_ids`

---

## 6. Neo4j 在 Phase 1 的角色

Phase 1 里 Neo4j 不需要太复杂。

最少只用来做：
- entity neighborhood lookup
- source relationship lookup
- claim relationship辅助查询

不要求一开始就做复杂 3-hop 图推理。

---

## 7. Phase 1 开发顺序

### Step 2
写 `alias_resolver`

### Step 3
写 `chunk_retriever`

### Step 4
写 `grounding_builder`

### Step 5
写 `answer_generator`

### Step 6
封成 `/qa` API

---

## 8. 验收标准

最少保证：

1. 能回答 20 条基础百科问题
2. 每个答案带来源 URL
3. 每个答案带 evidence chunk
4. 中文 query 能通过 alias 命中英文实体
5. 对 item / runeword / skill / area / monster 五类问题至少可用

---

## 9. Phase 1 不做什么

明确不做：

- 全量多跳推理
- 完整版本冲突治理
- 复杂 build 规划
- 自动持续增量抓取
- 全量中文知识闭环

这些属于后续阶段。

---

## 10. 当前最推荐的最小实现栈

- 数据入口：`docs/tier0/merged/`
- 向量库：`ChromaDB`
- 图数据库：`Neo4j`
- LLM：你后续自选

### 先做的最小主链路

```text
query
 -> alias_resolver
 -> canonical entity
 -> chunk retrieval (Chroma + lexical)
 -> grounding_builder
 -> answer_generator
```

---

## 11. 一句话建议

> 先把“能回答 + 有证据 + 有来源”的最小闭环做出来，不要一开始就做满配 GraphRAG。
