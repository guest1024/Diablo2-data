# Diablo II 快照与增量抓取维护手册

这份手册解释：

1. 当前站点抓取产物存在哪里
2. 什么叫“快照链接”
3. 后续如何做增量抓取与小更新

---

## 1. 当前抓取产物在哪

### 原始网页/接口抓取

- `docs/tier0/raw/<source-id>/`

例如：

- `docs/tier0/raw/arreat-summit/`
- `docs/tier0/raw/diablo2-io/`
- `docs/tier0/raw/blizzhackers-d2data/`

### URL 清单

- `docs/tier0/url-inventories/*.txt`

### 抓取执行记录

- `docs/tier0/fetch-manifest.json`
- `docs/tier0/fetch-report.md`

### Target 级抓取元数据

- `docs/tier0/raw-metadata/<source-id>/*.json`

这些文件记录单个抓取目标的：

- `url`
- `output_path`
- `sha256`
- `bytes_written`
- `content_type`
- `etag`
- `last_modified`
- `fetched_at`
- `fetch_mode`

### 快照清单

- `docs/tier0/snapshot-manifest.json`
- `docs/tier0/snapshot-manifest.md`

## 2. 什么是“快照链接”

在这个项目里，快照链接指的是：

- 原始 URL
- 对应本地静态文件路径
- 文件大小
- sha256
- 来源 source_id
- 目标级抓取元数据（若已生成）

也就是把：

```text
URL -> 本地快照文件
```

这个映射固定下来，方便以后：

- 重建数据
- 做增量更新
- 对比文件是否变化

---

## 3. 当前增量抓取策略

当前已经是 **cache-first + metadata-aware**：

- 如果目标文件已经存在，会优先复用本地快照
- 如果已经存在 target metadata，则抓取脚本可以记录并保留：
  - `etag`
  - `last_modified`
  - `fetch_mode`
- 这样重复执行时不会全量重抓

这意味着：

- 已抓过的静态页面不会反复下载
- 本地更新量通常较小

---

## 4. 下一步更好的增量抓取方向

当前已经完成的增量基础能力：

1. `raw/` 原始快照保存
2. `raw-metadata/` 目标级元数据保存
3. `snapshot-manifest` 快照清单构建
4. `verify_snapshot_manifest.py` 快照验证

后续建议继续增强为：

1. `sitemap lastmod` 检查
2. `etag / if-modified-since` 条件请求
3. `snapshot-manifest` 差异对比
4. 只重抓：
   - 新 URL
   - hash 变化 URL
   - lastmod 更新 URL

这样每次更新会更小、更快、更可控。

## 5. 当前推荐维护流程

### 重新生成快照索引

```bash
source .venv/bin/activate
python scripts/fetch_tier0.py --timeout 5
python scripts/build_snapshot_manifest.py
python scripts/verify_snapshot_manifest.py
```

### 看当前抓取覆盖

优先看：

- `docs/tier0/fetch-report.md`
- `docs/tier0/snapshot-manifest.md`
- `docs/tier0/raw-metadata/`

### 看当前是否可验证

```bash
source .venv/bin/activate
python scripts/verify_first_system_stack.py
```

---

## 6. 当前一句话原则

> 原始站点抓取产物保存在 `raw/`，目标级抓取状态保存在 `raw-metadata/`，URL 到本地快照的关系保存在 `snapshot-manifest`，后续增量抓取以“尽量少抓、只抓变化页面”为原则。
