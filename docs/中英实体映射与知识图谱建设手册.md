# Diablo II 中英实体映射与知识图谱建设手册

这份文档专门回答两个关键问题：

1. **怎么快速做好中文 / 英文实体与重要关系名词的映射**
2. **这个游戏知识库的图谱到底应该建立成什么样，才更适合百科搜索 / GraphRAG**

目标不是空谈，而是给你一个能直接落地到当前仓库的数据与工程方案。

---

## 1. 先讲结论

### 1.1 中英实体映射不要从“翻译”开始，要从“规范化主键”开始

正确顺序：

1. 先确定 **canonical entity**
2. 再挂载中英别名
3. 再挂载社区黑话 / 缩写 / 版本旧名
4. 最后再做 embedding / Graph / 检索优化

不要反过来：

- 不要先做翻译词典再想实体
- 不要直接把中文名 / 英文名当主键
- 不要把一个黑话词直接等同于一个页面标题

### 1.2 图谱不要一开始追求“超级复杂”，而要先做“检索友好型知识图谱”

第一版最重要的是：

- 实体可定位
- 别名可映射
- 来源可追溯
- 版本可区分
- 关系可扩展

所以图谱第一优先级不是“推理炫技”，而是：

- **百科搜索可用**
- **问答 grounding 可用**
- **关系扩展可控**

## 2. 问题一：怎么快速做好中文 / 英文实体映射

## 2.1 最快可落地的五层映射法

建议把映射体系拆成 5 层：

### 层 1：Canonical Entity 层

这是唯一主键层。

例子：

- `Runeword:spirit`
- `UniqueItem:harlequin_crest`
- `UniqueItem:stone_of_jordan`
- `Area:chaos_sanctuary`
- `Boss:mephisto`

要求：

- 语言无关
- 版本稳定
- 不能直接依赖页面标题

### 层 2：Alias Lexicon 层

这是标准别名层：

- 英文标准名
- 中文常用名
- 官方译名
- 页面 slug
- 页面 title

例子：

- Spirit
- 精神
- Spirit Runeword
- spirit-t3

### 层 3：Community Slang 层

这是最关键的一层，也是召回率提升最大的层：

- 军帽
- 超市
- 劳模
- 虫子
- 吉永
- 蛛网
- 安头
- 米山
- 锤丁

### 层 4：Abbreviation / Acronym 层

这是第二关键层：

- SOJ
- CTA
- COH
- BOTD
- HOTO
- MF
- USC
- ULC
- DClone

### 层 5：Contextual Mapping 层

这是为歧义准备的层。

同一个词，在不同上下文可能对应不同实体：

- `精神`
  - 可能是 `Spirit Runeword`
  - 也可能是 `Bone Spirit`
- `虫子`
  - 在 Diablo II 语境下常指 Andariel bug
  - 在别的游戏中就不是

所以这一层不能只做字符串映射，还要做：

- query context expansion
- candidate ranking
- source bias
- entity type bias

## 2.2 当前仓库里最快速的工程做法

你现在已经有两类关键资产：

### A. 词表层

- `docs/tier0/bilingual-term-map.json`

### B. curated anchor 层

- `docs/tier0/curated/documents.jsonl`
- `docs/tier0/curated/chunks.jsonl`

所以最实用的做法不是“重新设计一切”，而是：

### Step 1：继续扩 `bilingual-term-map.json`

每个 term 至少补：

```json
{
  "军帽": {
    "canonical_hint": "Harlequin Crest",
    "aliases": ["harlequin crest", "shako", "谐角之冠"],
    "preferred_source_ids": ["curated-anchor", "diablo2-io"]
  }
}
```

### Step 2：遇到高价值高频词，就补 curated anchor

适合补 curated anchor 的类型：

- 社区黑话
- 装备俗称
- build 简称
- 缩写
- 歧义严重但问得多的概念

### Step 3：把 alias map 变成“可维护资产”，不是 prompt 里的临时规则

推荐长期拆成：

- `term-map`：术语 -> canonical_hint
- `alias-registry`：实体 -> aliases[]
- `curated-catalog`：实体 -> 锚点卡 -> query coverage

### Step 4：引入最小实体归一流程

建议后续固定：

```text
query
-> normalize
-> slang expansion
-> acronym expansion
-> canonical_hint bias
-> candidate entity ranking
-> retrieve evidence
```

### Step 5：做“映射覆盖率”统计

后续每扩一轮，应该统计：

- 新增 term 数
- 新增 curated card 数
- 新增 routed query 数
- 热门 query 覆盖率

这件事你现在已经有基础了：

- `surface-coverage-report`
- `routing-matrix`
- `curated-catalog`
- `term-map-catalog`

## 2.3 如何更系统地自动抽中英映射

当前最优策略不是纯 LLM，也不是纯规则，而是 **规则优先 + LLM 辅助 + 人工确认入库**。

### 可用流程

#### 方法 A：标题/slug 规则抽取

从以下信息抽取：

- 页面标题
- URL slug
- canonical entity name
- item / skill / area 类型

优点：

- 稳
- 可复现
- 便于批处理

#### 方法 B：跨语言共同上下文抽取

当中文页面和英文页面都提到：

- 同样的特征属性
- 同样的位置
- 同样的掉落物
- 同样的技能描述

可把它们候选聚合为同一实体。

#### 方法 C：LLM 生成候选 alias

适合做：

- 社区黑话补全
- build 名补全
- 装备俗称补全

但生成后必须：

1. 进入候选区
2. 人工确认
3. 再写入 term map / alias registry

不要直接自动入生产词表。

#### 方法 D：query log 反推别名

将来上线后最有价值：

- 用户怎么问
- 当前召回到什么
- 哪些 query 失败

再用失败 query 反推：

- 需要加什么 alias
- 需要补什么 curated card

---

## 2.4 推荐你采用的“映射数据结构”

建议后续拆成三份：

### 1. `bilingual-term-map.json`

面向 query expansion

### 2. `alias-registry.jsonl`

面向 entity canonicalization

推荐结构：

```json
{
  "canonical_id": "UniqueItem:stone_of_jordan",
  "alias": "乔丹",
  "alias_type": "community_slang",
  "language": "zh",
  "confidence": 0.99
}
```

### 3. `term-equivalence.jsonl`

面向图谱里的跨语言 / 同义关系

推荐结构：

```json
{
  "lhs": "乔丹",
  "rhs": "Stone of Jordan",
  "relation": "same_as_term",
  "confidence": 0.99
}
```

## 3. 问题二：Diablo II 图谱到底应该长什么样

## 3.1 目标不是“通用大一统图谱”，而是“百科搜索友好图谱”

百科搜索 / GraphRAG 最常见的问题不是：

- 能不能做 7 跳推理

而是：

- 这个词到底是什么
- 它和哪些东西有关
- 它有哪些来源
- 是 LoD 还是 D2R
- 这个说法和另一个说法是不是一个东西

所以图谱设计要围绕这 5 个目标：

1. **entity resolution**
2. **alias resolution**
3. **source grounding**
4. **version awareness**
5. **relation expansion**

---

## 3.2 推荐图谱分层

建议做成三层图谱：

### Layer 1：Lexical / Alias Graph

节点：

- Term
- Alias
- Acronym
- Slang

边：

- `ALIASES`
- `SAME_AS_TERM`
- `ABBREVIATES`
- `COMMUNITY_REFERS_TO`

用途：

- 中文黑话 -> 英文实体
- 缩写 -> 标准术语
- 多个写法归一

### Layer 2：Entity Graph

节点：

- Runeword
- UniqueItem
- SetItem
- BaseItem
- Rune
- Recipe
- Skill
- Class
- Monster
- Boss
- Area
- Quest
- NPC
- BuildArchetype

边：

- `USES_RUNE`
- `REQUIRES_BASE`
- `DROPS_IN`
- `FOUND_IN`
- `BELONGS_TO_CLASS`
- `USES_SKILL`
- `USES_ITEM`
- `COUNTERS`
- `SYNERGIZES_WITH`
- `POPULAR_FOR`

用途：

- 百科关系扩展
- 构建实体邻域
- GraphRAG 子图召回

### Layer 3：Evidence / Provenance Graph

节点：

- SourceDocument
- Chunk
- Claim
- Patch

边：

- `MENTIONED_IN`
- `SUPPORTED_BY`
- `EVIDENCED_BY`
- `VALID_IN_PATCH`
- `CONTRADICTS`

用途：

- 来源追溯
- 多来源比较
- 版本过滤

---

### 核心实体节点

至少要有：

第一阶段只做最有价值的边：

## 3.5 图谱应该怎么服务百科搜索

建议把百科搜索做成两段：

### 阶段 A：实体定位

输入：

`军帽是什么`

系统先做：

`军帽 -> Harlequin Crest -> UniqueItem`

### 阶段 B：知识展开

再从实体图上取：

- alias
- type
- related build
- related classes
- related drops / area / recipe
- supporting sources

最后再去 chunks 里取证据文本。

也就是：

```text
query
-> alias graph
-> canonical entity
-> entity graph expansion
-> evidence graph grounding
-> text evidence retrieval
```

这才是最适合百科搜索的 GraphRAG。

## 3.6 通用知识图谱建设上，我们现在有没有更好的方法

有，但不是“更复杂”，而是“更分层”。

### 目前比传统单图更好的方法

#### 方法 1：Property Graph + Evidence Graph 双层法

比只做单一 entity graph 更好。

原因：

- 实体层负责关系
- evidence 层负责出处
- 避免把图谱做成大杂烩

#### 方法 2：Canonical Graph + Retrieval Overlay

比“把所有 chunk 都节点化”更好。

原因：

- 图节点只保留高价值结构化实体
 - retrieval overlay 保留大规模正文证据
 - 图和文本各自做最擅长的事

#### 方法 3：Alias Graph 先行，Entity Graph 后置

比“先做大图，再想召回”更好。

原因：

- 真实失败更多发生在实体映射层
- 别名、黑话、缩写如果没打通，后面的图结构再漂亮也没用
- 先把 alias / curated / canonical 做稳，GraphRAG 才会稳定

#### 方法 4：Curated Anchor + Open Corpus 混合

比“所有信息都只靠原始网页召回”更好。

原因：

- 社区黑话并不天然出现在英文资料页标题中
- 通过 curated anchor 可以把：
  - 中文俗称
  - 英文标准名
  - build archetype
  - 游戏上下文
  先对齐
- 再把原始网页当作二级证据层

这正是你当前仓库已经走通的路线。

## 4. 我建议你后续怎么建这个图谱

## 4.1 第一阶段：检索友好图谱

先不要追求复杂推理，先让下面这些事情稳定：

1. 中文俗称 -> canonical entity
2. 英文名 -> canonical entity
3. 缩写 -> canonical entity
4. entity -> source / chunk / claim
5. entity -> related entities

也就是先把图谱做成：

- **检索友好**
- **百科友好**
- **来源友好**

### 第一阶段的推荐 schema

#### Node

- `Entity`
- `AliasTerm`
- `SourceDocument`
- `Chunk`
- `Claim`
- `Patch`

#### Edge

- `ALIAS_OF`
- `SUPPORTED_BY`
- `EVIDENCED_BY`
- `VALID_IN_PATCH`
- `RELATED_TO`

#### 这阶段先不急着做的

- 复杂事件时序图
- 全量 chunk 节点互连
- 自动关系抽取到图的全自动闭环

---

## 4.2 第二阶段：游戏知识图谱

当实体映射层稳定后，再扩实体关系层。

### 最值得优先补的关系

#### 装备 / 符文之语方向

- `USES_RUNE`
- `REQUIRES_BASE`
- `SYNERGIZES_WITH_CLASS`
- `POPULAR_FOR_BUILD`

#### 怪物 / 场景方向

- `FOUND_IN_AREA`
- `DROPS_IN`
- `HAS_IMMUNITY`
- `HAS_ABILITY`

#### Build / 职业方向

- `USES_SKILL`
- `USES_ITEM`
- `USES_MERC`
- `FARMS_AREA`

#### 版本方向

- `VALID_IN_PATCH`
- `CHANGED_IN_PATCH`
- `CONTRADICTS`

---

## 4.3 第三阶段：GraphRAG 查询图

等图谱和 alias 层都稳后，再让它为问答服务：

### 图谱在问答里主要做 4 件事

1. **实体消歧**
2. **邻域扩展**
3. **版本限制**
4. **来源追踪**

### 推荐查询流程

```text
用户问题
-> term map expansion
-> curated anchor / alias resolution
-> canonical entity
-> graph neighborhood expansion
-> provenance + evidence chunk retrieval
-> rerank
-> answer generation
```

## 7. 你最关心的两个问题，最终一句话答案

### Q1：怎么快速做好中英文实体映射？

**答：不要先做翻译，而要先做 canonical_id，再做 alias lexicon、community slang、acronym 和 curated anchor 四层映射。**

### Q2：图谱应该建成什么样？

**答：先建“检索友好型知识图谱”，用 Alias Graph + Entity Graph + Evidence Graph 三层结构，让图谱先服务百科搜索、来源追溯和版本过滤，再逐步增强多跳关系与 GraphRAG 推理。**
