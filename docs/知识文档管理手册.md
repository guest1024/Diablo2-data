# Diablo II 知识文档管理手册

这份手册定义本项目文档体系的用途、边界和维护方式，目标是：

1. 文档不要分散到难以查看
2. 每份文档都有明确读者和用途
3. 数据、RAG、图谱、验证、运行方式都能快速定位

---

## 3. 每类文档应该写什么

### 总览文档

必须回答：

- 这个项目是做什么的
- 当前有什么
- 后续怎么用
- 应该先看哪几份文档

### 数据文档

必须回答：

- 数据从哪里来
- 数据是什么性质
- 当前覆盖哪些内容
- 以后如何增量维护

### 研发文档

必须回答：

- 架构为什么这样设计
- 当前缺口是什么
- 下一步怎么落地

### 验证文档

必须回答：

- 当前能力是否可验证
- 哪些 query / route 已通过
- 覆盖率有多少
- 当前是否回归通过

## 5. 文档维护规则

### 新增数据源时

至少更新：

- `docs/README.md`
- `docs/tier0/source-registry.json`
- 对应 source assessment / pipeline / verification 文档

### 新增术语 / alias / curated card 时

至少更新：

- `docs/tier0/bilingual-term-map.json`
- `docs/tier0/curated/`
- `routing-matrix`
- `surface-coverage-report`

### 新增图谱结构资产时

至少更新：

- `docs/tier0/graph-schema-v1.md`
- `docs/data-spec-v1.md`
- `docs/RAG开发手册.md`

## 6. 当前文档体系是否足够

当前已经具备：

- 项目总览
- 技术方案
- 数据规范
- 图谱方案
- 社区问答补强方案
- 用户启动文档
- 验证闭环文档

所以现在最缺的不是“再写很多零散文档”，而是：

> **把已有文档按角色和用途组织清楚。**

这也是这份手册存在的目的。
